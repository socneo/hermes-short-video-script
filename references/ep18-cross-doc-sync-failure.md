# EP18 v1.2 宣发文案P0未同步事故

**日期**：2026-06-20
**严重级别**：P0 硬伤

## 事故描述

EP18 v1.2 口播稿本体完成了3项P0级修订（"造假"→"注水"、"强制国标"→"准入技术要求"、"9中7"→"多名"），但宣发文案（抖音版、小红书版）和关键数据表中全部未同步。

## Tabbit 审核原话

> "口播稿本体中三项P0修订均已执行，但宣发文案和数据表未同步，形成'正文已改、周边未跟'的脱节现象，这是本稿最严重的问题。"

## 未同步清单（共6处）

| 位置 | 应改为 | 仍残留 |
|------|--------|--------|
| 节奏表 | FSD数据**注水**拆解 | FSD数据**造假**拆解 |
| 抖音文案 | FSD数据**注水**被扒 | FSD数据**造假**被扒 |
| 小红书标题 | FSD数据**注水**被扒 | FSD数据**造假**被扒 |
| 小红书正文 | **多名**前员工 | **9个前员工中7个** |
| 数据表 | **多名**前员工不信任 | **前员工9人受访7人不信任** |
| 小红书正文 | 工信部把激光雷达列入L3**准入技术要求** | 工信部把激光雷达列进L3**强制国标** |

## 根因分析

1. **审核流程缺陷**：口播稿定稿后，只检查了正文是否修改到位，没有执行"宣发文案逐条扫描"
2. **同步检查清单存在但执行不到位**：skill中已有"口播↔宣发↔数据表三方同步"检查项，但该检查项在v1.2定稿时未执行
3. **误区**：认为"口播稿是主文档，宣发文案是附属"——实际上宣发文案是独立传播物，法律风险反而更高（抖音/小红书平台直接审核文案内容）

## 修正措施

v1.3 逐条扫描后全部同步：
- 节奏表 ✅ 
- 抖音文案 ✅
- 小红书标题+正文 ✅
- 数据表 ✅

## 教训

1. **P0修订 = 全文同步**：任何P0级修订（法律术语、事实数据、品牌安全）必须在口播稿+宣发抖音+宣发小红书+数据表中**全文扫描替换**
2. **"✅ 可录制"标记 = 三方全部完成**：只有口播稿改完就标"可录制"是错误操作
3. **审核顺序**：口播稿定稿 → 宣发文案同步 → 数据表同步 → 通读确认 → 标记"可录制"
4. **修订记录中也要标注**：P0修订项要在修订记录表中注明"含宣发文案同步"
